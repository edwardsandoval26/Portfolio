## Problem Statement

Financial fraud costs the global economy over $5 trillion annually. Traditional rule-based systems catch only 40-50% of fraudulent transactions and generate high false-positive rates, frustrating legitimate customers and overwhelming review teams.

## Dataset

- **Source**: Synthetic financial transaction dataset (based on PaySim simulator)
- **Size**: 6.3M transactions over 30 days
- **Features**: 12 raw features including transaction amount, type, origin/destination balances
- **Class imbalance**: 99.87% legitimate vs 0.13% fraudulent

## Motivation

Banks need a system that can score transactions in under 100ms while maintaining high precision. The goal was to build a production-grade system that could handle 10,000+ transactions per second with minimal false positives.

## Data Preprocessing & Feature Engineering

Key engineered features that improved model performance:

- **Velocity features**: Transaction count and volume in rolling 1h, 6h, 24h windows
- **Behavioral deviation**: Z-score of transaction amount relative to user's historical mean
- **Network features**: PageRank score of sender in the transaction graph
- **Time-based**: Hour of day, day of week, time since last transaction

```python
def compute_velocity_features(df, windows=[1, 6, 24]):
    for w in windows:
        df[f'tx_count_{w}h'] = df.groupby('sender_id')['amount'].transform(
            lambda x: x.rolling(f'{w}H').count()
        )
        df[f'tx_volume_{w}h'] = df.groupby('sender_id')['amount'].transform(
            lambda x: x.rolling(f'{w}H').sum()
        )
    return df
```

## Model Architecture

After benchmarking Logistic Regression, Random Forest, and XGBoost:

| Model | Precision | Recall | F1-Score | Latency (p99) |
|-------|-----------|--------|----------|---------------|
| Logistic Regression | 0.72 | 0.65 | 0.68 | 2ms |
| Random Forest | 0.89 | 0.78 | 0.83 | 15ms |
| **XGBoost** | **0.94** | **0.87** | **0.90** | **8ms** |

Final model: XGBoost with 500 trees, max_depth=6, trained with focal loss to handle class imbalance.

## Evaluation & Results

- **AUPRC**: 0.92 (Area Under Precision-Recall Curve)
- **False Positive Rate**: 0.3% (down from 2.1% with rule-based)
- **Detection Rate**: 87% of fraud caught in real-time
- **Latency**: p50=3ms, p99=8ms

## Engineering Details

- **Serving**: FastAPI with async endpoints, ONNX runtime for inference
- **Streaming**: Kafka for real-time transaction ingestion
- **Feature Store**: Redis for low-latency feature retrieval
- **Monitoring**: Prometheus + Grafana for model performance dashboards
- **Deployment**: Docker containers on AWS ECS with auto-scaling
- **Experiment Tracking**: MLflow for model versioning and comparison
- **CI/CD**: GitHub Actions for automated testing and deployment

## Lessons Learned

1. Feature engineering contributed more to performance than model selection
2. ONNX conversion reduced inference latency by 60% compared to native XGBoost
3. Proper handling of class imbalance (focal loss + SMOTE) was critical
